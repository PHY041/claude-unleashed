It determines what Claude can do without asking, what requires your explicit approval, and what is categorically denied regardless of any other settings. Most users think of it as a binary. Permission is either on or off. Ask or don't ask. That's not what it is. The permission system is a multi-layer decision tree that evaluates every operation Claude wants to perform against a set of rules sourced from up to eight different places, processed through three distinct phases, with the final answer shaped by which of six operational modes you've chosen, with a speculative pre-execution engine running in parallel when possible. That sounds complex. And it is. But the complexity serves a genuine purpose: it means the system can express nuanced, situation-specific policies. "Allow all git commands, but not git push. Allow file edits in the src/ directory, but not in the config/ directory. Allow reading anything, but ask before running any npm commands, and deny outright any rm command that could be destructive." That kind of policy is expressible, and once expressed, it runs silently, correctly, every single time. The developers who learn the permission system deeply stop thinking of it as a gate to get past. They think of it as a specification of their working relationship with Claude — a contract that says "here's what I'm comfortable with happening automatically, here's what needs my judgment, and here's what should never happen regardless of context." Let's build that understanding, starting with the six modes. ### The Six Permission Modes **Mode 1: default** Standard interactive mode. Claude asks for permission before every file modification and every shell command it hasn't already been explicitly told it can run. This is the right mode when: you're exploring a new codebase and want to stay aware of every change Claude makes. You're doing maintenance work on someone else's project. You're at the beginning of a task whose scope you're not confident about. You're a less technical user who wants to stay informed about what's happening to your files. The tradeoff is deliberate. Every operation that isn't on an explicit allow list surfaces for review. For a developer who's careful about unfamiliar territory, this is protective. For a developer who's iterating rapidly on their own project, this is friction. The scenario that justifies default mode: Marcus is a backend developer who has just inherited maintenance responsibility for a client's payment processing system. He wasn't involved in building it. He's read the documentation but hasn't worked in the codebase. When Claude wants to edit `src/payment/processor.ts`, Marcus wants to see exactly what change is proposed before it gets made — not because he doesn't trust Claude, but because he doesn't yet know what a "normal" change to that file looks like. When Claude wants to run a database migration script, he absolutely wants to review it first. Default mode gives him that oversight. The friction is the point. The cost of default mode isn't just your time. It's your attention. And attention is the scarce resource. When every operation requires a click, you either read every prompt carefully (protective but exhausting) or you start clicking reflexively (fast but providing no safety). The goal of good permission configuration is to route only the operations that warrant genuine attention to your attention — and let everything else proceed automatically. **Mode 2: acceptEdits** Auto-approves file edits within the current working directory. Still asks for shell command permissions. This is the right mode for: active development sessions where you're iterating fast on a known codebase. You know the project. You asked Claude to make changes. The "do you want Claude to edit this file" prompt isn't providing useful information — you just said "edit this file." What changes: file writes in the current working directory happen automatically. Claude edits a React component, creates a test file, modifies a CSS stylesheet — all without prompting. Shell commands still surface for your review, because shell commands have a wider range of potential effects. The scenario: Layla is building a new API endpoint. She's going to ask Claude to create four new files, modify two existing service files, and write tests. The file edits are not what she wants to review — she'll review the results when they're done. What she does want to see is the shell commands, because Claude sometimes adds flags or options to commands that she wants to examine. `acceptEdits` gives her exactly that. File writes happen automatically. Shell commands surface for review. In a typical session, this eliminates about 80% of permission prompts, and the remaining 20% are the ones that actually warrant a look. Note the boundary: "within the current working directory." If Claude tries to edit a file outside the directory where you launched Claude Code, `acceptEdits` doesn't apply. The operation asks for permission. This is deliberate — the working directory boundary is a reasonable proxy for "files I'm working on right now," and anything outside that boundary is more likely to be consequential. There's a deeper reason that `acceptEdits` stops at the current working directory boundary. When you launch Claude Code from a directory, you're implicitly scoping the session to that project. You're saying "this is the context we're working in." Files outside that directory belong to different contexts — other projects, system configuration, personal files — and operations on them warrant the extra moment of consideration that a permission prompt provides.
