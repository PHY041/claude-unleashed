# Appendix B: 80+ Feature Flags Decoded > The hidden control panel. What the flags are, what they unlock, and what they tell you about where Claude Code is going. --- ## Introduction: What Feature Flags Are and Why They Matter Every time Claude Code starts, it loads a configuration from Anthropic's GrowthBook feature flag system. These flags are remote-controlled switches that can turn features on or off for specific users, time windows, or rollout percentages — without shipping a new version of the application. This is a standard practice in modern software development. What makes Claude Code's flag system unusual is its scale (80+ flags), its specificity (individual sub-features of major systems are independently flagged), and what happens to flagged code in the compiled binary. ### How to Detect Which Flags Are Enabled in YOUR Installation The quickest way to see which flags your installation has active is to run Claude Code with debug logging: ```bash CLAUDE_CODE_DEBUG=flags claude ``` This outputs all flag values at startup. A typical external user will see the default-on flags returning `true`, and everything else returning `false` or not appearing at all. You can also probe specific behavior: - `/voice` — if VOICE_MODE is enabled, this command exists; otherwise it doesn't - `/buddy` — if BUDDY is enabled, shows your companion card; otherwise "command not found" - `/advisor` — should work for all external users (ADVISOR is default on) - `USER_TYPE` check: `echo $USER_TYPE` — if empty, you're an external user; `ant` means Anthropic employee access For the most thorough detection, inspect the npm source map: ```bash npm install @anthropic-ai/claude-code # Find the source map reference in the bundle grep -r "feature\(" node_modules/@anthropic-ai/claude-code --include="*.js" | head -20 ``` This will show all `feature('FLAG_NAME')` calls in the bundle. Any flag where the call is replaced by `false` (a literal) has been compiled out — that feature genuinely doesn't exist in your binary. ### Bun Dead Code Elimination — The Mechanism in Detail Claude Code is built with Bun, a JavaScript runtime and bundler that performs dead code elimination at compile time. This is the key to understanding why feature flags in Claude Code are fundamentally different from feature flags in most web applications. In a typical web application, a feature flag is a runtime check: ```typescript // Typical web app (runtime): if (await featureFlags.get('NEW_FEATURE')) { renderNewFeature() } else { renderOldFeature() } ``` Both code paths exist in the binary. The flag determines which executes. An attacker who could manipulate the flag could enable disabled features. In Claude Code, the flags are evaluated at build time: ```typescript // Claude Code source: if (feature('BUDDY')) { // 45,000 lines of companion pet system initializeBuddySystem() renderBuddySprite(...) runGachaRoll(...) } // Compiled output for external users (BUDDY = false at build time): // (literally nothing here — the entire block is gone from the binary) ``` When `feature('BUDDY')` evaluates to `false` at build time, Bun's bundler treats the `if` body as dead code and removes it entirely from the output. This is the same optimization that minifiers have done for decades with `if (false) { ... }` — it's just automated through the feature flag system. The security implication: **you cannot enable a compiled-out feature by manipulating environment variables or config files.** The code doesn't exist in your binary. Setting `BUDDY_ENABLED=true` in your environment does nothing, because there is no BUDDY initialization code to call. You would need Anthropic to compile and distribute a new binary with the flag set to `true` at build time. This is why the feature flag system is worth understanding: it's an archaeological record of what Anthropic is building, not a configuration mechanism external users can manipulate. **Historical examples of flags that shipped:** - `COMPACT` — Context compaction was once a feature flag. It graduated to always-on when the team was confident it worked correctly for all users. - `MEMORY` — The CLAUDE.md memory system was once opt-in. It shipped as default-on when the team validated its behavior across diverse use cases. - `HOOKS` — Lifecycle hooks were an advanced feature before being promoted to a core capability. - `SKILLS` — Custom slash commands via Markdown files went through the same progression. The timeline from ANT-only to general availability varies. `COMPACT` took approximately 4 months from ANT-internal to default-on. `VOICE_MODE` has been code-complete for longer — the bottleneck is infrastructure (Anthropic's STT WebSocket endpoint), not the code itself. ### Why Anthropic Uses This Approach 1. **Progressive rollout:** Ship to 1% of users, verify stability, expand to 10%, 50%, 100%. Each step is a flag change in GrowthBook, not a deployment. If problems appear at 10%, roll back the flag without touching the binary. 2. **A/B testing:** Test whether users who get feature X retain longer, use Claude Code more, report fewer bugs. Measure, then decide. Features that don't improve measurable outcomes get cut. 3. **ANT-only dogfooding:** Anthropic employees use Claude Code every day to build Claude Code. They get access to every feature so the team can experience unfinished capabilities in their real workflow. Bugs found internally are less costly than bugs found by external users. 4. **Emergency kill switch:** If a shipped feature causes problems in production, flip the flag to `false` in GrowthBook. Existing binaries stop executing that code path immediately. No deployment, no downtime. ### Reading Flags as a Product Roadmap The flags in Claude Code's source tell you what Anthropic is building. Features that are complete but flagged off (like VOICE_MODE, BUDDY, and WEB_BROWSER_TOOL) are closer to shipping than features that are partial implementations. Features that are ANT-only are further out. Features that have no flag (shipped fully) used to be flagged, then graduated. The standard progression: `stub (constant defined, no code) → partial implementation (some code exists) → complete (full implementation, just disabled) → ANT-only (complete + enabled for Anthropic staff) → opt-in beta → default on → core (flag removed entirely)` --- ## Flag Categories ### User-Facing Features (Currently Enabled by Default) These flags are `true` for all external users.
