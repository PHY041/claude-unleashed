# Appendix A: Complete settings.json Reference > The power user's most important file. Every field, every option, every hook — documented in full. --- ## Introduction: Why settings.json Is Everything Most Claude Code users interact with it through the command line. They type, Claude responds, files change. It works. But somewhere around week two, you start noticing friction: the same permission dialogs appearing for every `git commit`, Claude asking whether it can read a file you've already said yes to a hundred times, or a shared codebase where everyone on the team has different opinions about what Claude should and shouldn't do. The solution to all of this is `settings.json`. This file is Claude Code's configuration layer. It tells Claude Code what it's allowed to do without asking, what it should never do, how to behave in different contexts, which tools to activate, and what to run automatically when certain events happen. A well-crafted `settings.json` can eliminate 90% of the permission dialogs in your daily workflow. A poorly crafted one can accidentally grant Claude more latitude than you want — or lock it down so completely it can't do anything useful. The settings system exists because different contexts need different behaviors. A solo developer on a personal project has different needs than a team of ten working on a production SaaS. A non-technical user needs Claude to ask before taking action; a senior engineer who's used Claude for months wants it to just do the thing. Settings accommodate all of these — not by picking one right way, but by giving you the tools to configure the right behavior for your situation. ### Where Does settings.json Live? Claude Code looks for settings in two places, and they serve different purposes: **Global settings** — `~/.claude/settings.json` This file applies to every project you work on. Put things here that you want to be true universally: your preferred model, your notification preferences, tools you always want available, and permission rules that apply across all your work. If you prefer dark theme, your global settings set it once and you never think about it again. **Project settings** — `.claude/settings.json` (in your project root) This file applies only when you're working in that specific project directory. It's for project-specific rules: "this project uses a PostgreSQL database, never drop tables," "this is a production service, require confirmation before any deployment commands," "this team uses conventional commits, always format commit messages this way." Project settings are typically committed to version control so the whole team gets the same behavior. ### Priority Order When Claude Code loads settings, it merges them in a specific priority order. Higher-priority settings win when there's a conflict: ``` policySettings (enterprise admin settings — highest priority) projectSettings (.claude/settings.json in current project) localSettings (.claude/.local/settings.json — machine-specific, gitignored) userSettings (~/.claude/settings.json — your global config) cliArg (--allow / --deny flags passed on the command line) ``` In practice: **project settings override global settings.** If your global settings say "never ask about git operations" but a specific project says "always ask about git push," the project wins. This lets teams enforce safety rules even for developers who have permissive personal configurations. The `.claude/.local/` directory is worth noting: it's intended for machine-specific overrides that shouldn't be committed to version control. If you're working on a project where one developer has different local tools than the rest of the team, local settings let them customize without affecting the shared configuration. Local settings are gitignored by default — they're the right place for secrets, personal API keys in the `env` section, or developer-specific preferences. The priority cascade also means enterprise-managed settings always win. If your company deploys a `policySettings` file (via MDM or a shared config management system), developers cannot override those rules in their personal settings. This is the mechanism for enterprise compliance: set it at the policy level and it sticks. --- ## Top-Level Fields ### `model` **Type:** `string` **Default:** `"claude-sonnet-4-5"` (current default at time of writing) **Source:** `src/utils/config.ts` Controls which Claude model Claude Code uses for responses. This is the most impactful performance-vs-cost tradeoff in your configuration. Every API call in your session uses this model unless explicitly overridden with `/model`. Available options: - `"claude-haiku-4-5"` — fastest, cheapest, ideal for simple tasks and exploration - `"claude-sonnet-4-5"` — balanced default, excellent for most development tasks - `"claude-opus-4-5"` — most capable, best for complex reasoning and architecture decisions **Before and After scenario:** *Without configuring `model`:* You're using the default, which is Sonnet. This works fine for most tasks. But you're refactoring a complex legacy codebase that requires deep multi-file understanding, tracking subtle dependencies, and reasoning about historical design decisions. Sonnet makes mistakes — misses a dependency here, suggests a pattern that won't work there. You spend time correcting errors that Opus would have avoided. *With `"model": "claude-opus-4-5"` in the project settings:* Claude Code reads the project-level config when you enter this directory. Opus handles the complexity. The refactoring suggestions are architecturally sound. You trust the output enough to review rather than rewrite. Your session takes 20% longer (Opus is slower) but you ship the refactoring in one afternoon instead of three. **Example:** ```json { "model": "claude-sonnet-4-5" } ``` **When to use it:** Set this at the project level if certain projects require heavy reasoning (set to Opus) or if you're doing rapid exploratory work where response time matters more than depth (set to Haiku). Many power users keep Sonnet globally and override to Opus for architecture sessions. **Cost implications:** See Appendix F for full pricing. Switching from Sonnet to Haiku can reduce costs by roughly 75% for input tokens. Switching from Sonnet to Opus increases costs 5x. The model choice affects every single API call in your session. **Common mistakes:** Setting Opus globally. Opus is 5x more expensive than Sonnet and noticeably slower. Most tasks don't need it. A better approach: keep Sonnet as your global default, override to Opus in project settings for projects that genuinely need it. Another common mistake: forgetting that the `model` field in project settings overrides global. If you upgrade globally to a newer model, projects with explicit `model` fields stay on the old one. **Interactions:** The `model` field interacts with `maxTokens` — some models have different maximum output token limits. Haiku and Sonnet support up to 64,000 output tokens; Opus supports up to 32,768. If you set `maxTokens: 64000` in a config that's used for Opus, Claude Code will silently cap it at the model's limit. --- ### `maxTokens` **Type:** `number` **Default:** `32768` **Source:** `src/utils/config.ts`, `src/utils/modelCost.ts` Sets the maximum number of output tokens Claude can generate in a single response. The practical ceiling is model-dependent. **Before and After scenario:** *Without configuring `maxTokens`:* You ask Claude to write a comprehensive test suite for a 300-function API layer. It generates the first 80 tests and stops mid-function with "I'll continue in the next response." You prompt it to continue. It generates more, stops again. This continues across five separate responses. Each continuation forgets some context about the previous batch, leading to duplicate test names and inconsistent patterns. What should have been one cohesive file is now a patchwork. *With `"maxTokens": 64000`:* Claude generates the entire test suite in one pass. Naming is consistent. Patterns are uniform. The file is complete. **Example:** ```json { "maxTokens": 64000 } ``` **When to use it:** Increase this when Claude needs to generate very long files — a 500-line Python module, a comprehensive test suite, a detailed architecture document. The default of 32,768 is often sufficient for typical development tasks (function implementations, explanations, short files), but if Claude keeps cutting off mid-generation with continuation prompts, bumping this value eliminates that friction. **Common mistakes:** Setting `maxTokens` to 64000 globally without understanding the cost implications. Output tokens are 5x more expensive than input tokens. A response that's twice as long costs twice as much in output. The default is set where it is because most responses don't need to be longer. Set this at the project level for specific projects that require long output, not globally. **Interactions:** `maxTokens` and `compactThreshold` both affect session length in different ways. `maxTokens` controls output length per response; `compactThreshold` controls when the total context gets compressed. Increasing both makes sessions "bigger" but also more expensive. --- ### `temperature` **Type:** `number` (0.0 to 1.0) **Default:** `1.0` **Source:** `src/utils/config.ts` Controls the randomness of Claude's responses. Lower values produce more deterministic, focused output. Higher values produce more creative, varied output. **Before and After scenario:** *Without configuring `temperature` (default 1.0):* You're generating database migration scaffolding — the same pattern, repeated 20 times with different field names. Each generation has slightly different formatting, slightly different variable naming, subtle inconsistencies in how null checks are expressed. You spend time normalizing the output. *With `"temperature": 0.2` in the project's `.claude/settings.json`:* The scaffolding is consistent. Same pattern every time. The low temperature removes the variation that the model would otherwise introduce. The output is boringly uniform, which is exactly what you want from scaffolding. **Example:** ```json { "temperature": 0.7 } ``` **When to use it:** For code generation and technical tasks, the default of 1.0 works well — Claude's instruction-following keeps it on-task even at high temperature. Lower this to 0.2-0.5 for tasks where consistency matters above all else: generating templated code, filling in scaffolding, or producing structured data that must follow a strict format. Keep it at 1.0 or higher for creative writing, brainstorming, or documentation where variety is welcome. **Common mistakes:** Setting temperature to 0 and expecting that to eliminate all non-determinism. Temperature 0 produces highly deterministic outputs for simple tasks, but complex tasks still have variation because the model's internal computation isn't a simple argmax. Also: setting low temperature globally when you use Claude for creative tasks too. This is a case for project-level overrides. **Interactions:** Temperature interacts with the model choice. Haiku at temperature 0.2 produces very predictable output but may struggle with complex reasoning tasks that require exploring multiple approaches. Opus at temperature 1.0 may produce varied output but the quality floor is higher. Generally: use temperature to tune predictability within a model tier, not to compensate for model capability. --- ### `permissions` **Type:** `object` **Default:** `{ "allow": [], "deny": [] }` **Source:** `src/types/permissions.ts`, `src/utils/permissions/permissions.ts` The permission system is the most important section of settings.json for daily workflow. It pre-approves (or pre-denies) operations so Claude Code doesn't have to stop and ask. The `permissions` object has two arrays: - `allow` — operations that are automatically approved without confirmation - `deny` — operations that are automatically blocked without asking **Before and After scenario:** *Without configuring `permissions`:* You've asked Claude to fix 15 failing tests. The first fix involves reading three files. Claude asks: "Can I read `src/utils/auth.ts`?" You click Allow. Then: "Can I read `src/utils/tokens.ts`?" Allow. Then: "Can I edit `tests/auth.test.ts`?" Allow. Then: "Can I run `npm test`?" Allow. You've clicked through four dialogs before anything has happened. Multiply this by 15 fixes. You spend half the session in dialog boxes. *With a configured `permissions.allow` list:* Read, Edit, and `Bash(npm *)` are pre-approved. Claude reads all three files, runs the tests, makes the edits, and runs tests again — all without interruption. You review the completed work rather than approving each step. **Rule syntax:** ```json { "permissions": { "allow": [ "Read", "Edit", "Write", "Bash(git *)", "Bash(npm *)", "Bash(node *)", "WebFetch(domain:*)", "WebSearch", "mcp__github" ], "deny": [ "Bash(rm -rf *)", "Bash(sudo rm *)", "Bash(curl * | bash *)" ] } } ``` **Rule formats:** - `"Read"` — allows the entire Read tool (all file reads) - `"Edit"` — allows the entire Edit tool (all file edits within CWD) - `"Bash(git *)"` — allows any git command (glob pattern) - `"Bash(git commit -m *)"` — allows only git commit with a message - `"mcp__github"` — allows all GitHub MCP tools - `"mcp__github__list_issues"` — allows only the specific list_issues MCP tool - `"WebFetch(domain:github.com)"` — allows fetching from a specific domain only **The eight rule sources, from highest to lowest priority:** 1. `policySettings` — enterprise admin (read-only) 2. `flagSettings` — feature flag rules 3. `projectSettings` —
