He drinks instant coffee from a ceramic mug he bought at a second-hand market in Hongdae, the kind of mug with a slight chip on the rim that you keep using because everything about it is exactly right except the chip. He has one monitor, a 27-inch LG he picked up refurbished from a Naver classifieds post for 180,000 won, and a mechanical keyboard that his neighbors, if they were the confrontational type, could reasonably complain about through the walls. In November 2025, Daniel was working a contract job maintaining an internal dashboard for a mid-size logistics company in Seoul. The work paid decently — better than his previous contract, which had involved a lot of legacy PHP that he'd rather not think about. He was billing around sixty hours a month and spending the rest of his time the way he usually spent it: poking at tools, building small side projects, abandoning most of them. His GitHub was full of repositories with encouraging names — `meal-planner-v2`, `budget-tracker-rewrite`, `study-scheduler` — and no recent commits. He had been using Claude Code for three months at that point. He had picked it up on a recommendation from the Claude Code community forum, started using it on the logistics dashboard work, and very quickly become the kind of user who couldn't imagine going back. He used it every day. He used it for everything that could reasonably be described as "figuring out code." He was, by his own description, "addicted to it in a productive way." And every day, he hit the same friction. The logistics company ran their entire project management operation on Jira. This is not unusual — half the engineering teams on earth run on Jira — but it created a specific, repetitive pain. Daniel's workflow looked like this: he would be deep in a codebase with Claude Code, debugging something or building a new feature, and he would need to understand what the Jira ticket was actually asking for. So he would alt-tab to his browser. He would navigate to Jira. He would find the ticket. He would read the acceptance criteria. He would read the comment thread. He would copy the relevant parts — the acceptance criteria, the linked tickets, the comment from the product manager clarifying a requirement, the note from QA about a specific test case they wanted to see — and he would paste them into his Claude Code session. Then he would alt-tab back. Context switch complete. He did this fifteen times a day. Sometimes twenty. The context switching was bad enough on its own — every time he left the flow state in Claude Code, he lost something, some thread of context, some momentum. But the information problem was worse. By the time he copied a comment thread into Claude, someone had often added a new response to it. The product manager would clarify something. A QA engineer would flag a concern. The information in his Claude Code session was always slightly stale, always a snapshot of a conversation that was still ongoing. And every time he wanted Claude to update a ticket — log a comment about what he'd done, change the status to "in progress," link a related issue he'd discovered — he had to switch back to Jira and do it manually. Claude would say "I've fixed the payment validation issue." Daniel would switch to Jira, find the ticket, add a comment saying he'd fixed the payment validation issue, and switch back. Two systems that should be in conversation with each other, connected only by his own hands. "I kept thinking, this is insane," he told me later, during a video call where I could see his desk setup in the background — the monitors, the mechanical keyboard, the instant coffee. "Claude can do so many things. It can read my files, run my tests, make my git commits. But I have to manually tell it what the Jira ticket says? It's 2025. The year is 2025 and I'm copying and pasting from a project management tool to my AI assistant like it's 2012." Daniel had heard about the Model Context Protocol. He had read the Anthropic documentation — the spec was clear enough. He understood the concept. He had seen people in the Discord talking about MCP servers they'd built. He had just never built one himself. The barrier felt higher than it probably was. Building a protocol server sounded like something that required weeks of work and a deep understanding of distributed systems. He decided to try anyway. --- ### What the Server Actually Does What the Jira MCP server actually does is more interesting than "connects Claude to Jira." That description makes it sound like a synchronization tool, a two-way data pipe, something in the infrastructure category that you deploy and forget about. What it actually does is something more fundamental: it makes Jira data a native part of Claude Code's cognitive context. The distinction matters. Synchronization tools move data from one place to another. What the MCP server does is give Claude the ability to ask questions. Claude can now formulate a question — "what does JIR-4821 say the acceptance criteria are?" — and get an answer, without any human in the loop. The data doesn't have to be pre-loaded or pre-synced. Claude asks for what it needs, when it needs it. When Daniel's server is running and configured, Claude Code can do things it couldn't do before. It can notice that you're working on the checkout flow, query the linked Jira ticket, and say: "The acceptance criterion says payment failures should display an error code. Your current implementation shows a generic error message without the code. Do you want me to add that?" Claude is reading the ticket in real time, correlating it with the code, and proactively identifying the gap. The developer doesn't have to think to ask. The AI asks for them. It can also do things in the other direction. When Claude finishes fixing a bug, instead of the developer having to switch to Jira and update the ticket, Claude can say "I've fixed the payment error — should I update the Jira ticket to show this is complete?" The developer says yes. Claude posts a comment with a summary of what it changed, links the relevant commit, and moves the ticket to "Done." The specific operations Daniel exposed in the initial version: **get_issue**: Retrieves a Jira issue by key. Returns the full data set — summary, description, status, priority, assignee, reporter, sprint assignment, story points, labels, components, all comments with their timestamps and authors, all linked issues with their relationship types, and all custom fields the team has defined. This is intentionally comprehensive. Daniel found early on that leaving out custom fields was the number-one source of complaints — every team had critical information in custom fields. **search_issues**: JQL-powered search. JQL (Jira Query Language) is the query syntax Jira uses natively — things like `project = "Backend" AND status = "In Progress" AND assignee = currentUser()`. Claude can formulate JQL queries, run them, and get back lists of matching issues. This turned out to be more powerful than individual ticket retrieval for many workflows — instead of "what does ticket X say," Claude could now answer "what are all the open bugs in this sprint that I should know about while I'm working on this feature." **update_issue**: Changes the status, assignee, priority, or other fields of an existing issue. Also adds comments. The status transitions respect Jira's workflow rules — if you're trying to move a ticket from "Open" to "Closed" in a workflow that requires an intermediate "In Review" state, the server handles the error gracefully and tells Claude what the valid transitions are. **get_sprint**: Retrieves all issues in the current active sprint for a given board. Returns a summarized view of sprint state — what's complete, what's in progress, what's blocked. Claude can use this as background context when asked "what's the state of the team right now?" or "is there anything I should prioritize over this feature work?" **link_issues**: Creates a link between two Jira issues. When Claude identifies that a bug it's fixing is related to another open issue, it can create the link directly. This turned out to be more valuable than expected — Claude would identify relationships between issues that the team hadn't explicitly noted, and the links made the project history more useful. **add_comment**: Posts a comment to a ticket. Claude can document what it did, what it discovered, what decisions it made — directly in the project management system where the rest of the team will see it. This closes the loop on developer communication that often gets lost: the "oh by the way, this fix required us to change the database schema slightly" information that should be in the ticket but usually ends up in a Slack message that nobody can find six months later. That's six operations. Not large. Not trying to replicate all of Jira. Just the operations that made the constant context switching unnecessary. It took Daniel two weeks to build, mostly in evenings after his contract work. The authentication part — OAuth 2.0 with Jira's API, handling token refresh, managing credentials without storing them in plaintext — took longer than the actual MCP implementation. The MCP part itself was, he says, surprisingly straightforward. "The protocol is really well designed," he said. "Once I understood that I was basically just writing JSON schemas for functions and then implementing those functions, it clicked. You describe what the function does, you describe the inputs, and you implement the function. It's not magic. It's just JSON-RPC with good tooling around it." --- ### What It Took to Build the Server The technical work itself deserves some attention here, because the gap between "building an MCP server sounds hard" and "building an MCP server actually is hard" matters for anyone who is deciding whether to try this. Daniel started with the TypeScript MCP SDK, published by Anthropic. The SDK is well-documented and maintained — this is the official implementation, not a community project. The package is `@modelcontextprotocol/sdk` on npm and the docs are at modelcontextprotocol.io. The first hour was understanding the protocol. JSON-RPC 2.0 is a lightweight protocol for remote procedure calls — you send a JSON request object with a method name and parameters, and you get back a JSON response with a result or error. It's simpler than REST in some ways and more structured in others. The MCP spec layers on top of this with specific method names (`tools/list`, `tools/call`, `resources/list`, `resources/read`, `prompts/list`, `prompts/get`) and specific data shapes for each. The second few hours were understanding Jira's REST API. Jira Cloud has a well-documented REST API. The authentication options are: API tokens (simple, personal), OAuth 2.0 (complex, can act as any user), and OAuth 2.0 server-to-server (complex, for service accounts). Daniel started with API tokens because it was faster and switched to supporting all three for the enterprise version. The actual MCP implementation — the code that connected the two — was the fastest part. Once he understood the protocol and the API, translating between them was relatively straightforward.
