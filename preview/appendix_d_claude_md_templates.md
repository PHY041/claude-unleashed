# Appendix D: CLAUDE.md Templates > Ready-to-use templates for every context. Start here; customize as you learn what Claude needs to know about your work. --- ## Introduction: The Compounding Value of CLAUDE.md Here's a calculation worth making. Suppose you work with Claude Code for 100 sessions over three months. In each session, without a CLAUDE.md, Claude has to ask four clarifying questions before it can do useful work: What kind of project is this? What's your preferred style? What should I not touch? What framework are you using? That's 400 clarifying exchanges. Each one adds roughly 30 seconds of waiting time and a small amount of friction. That's 200 minutes of overhead — over three hours of your life spent answering the same questions. A well-written CLAUDE.md eliminates most of those questions from session one. Three minutes to write it once; three hours saved over the quarter. The compounding works in your favor: every session, every agent spawn, every time Claude reads that file, you get the benefit. But the bigger benefit is harder to quantify. When Claude knows your context, it doesn't just save you from clarifying questions — it makes better decisions. It knows not to suggest the one library you hate. It knows your test requirements before it starts writing code. It knows that "quick fix" means "don't rewrite the whole module." Context makes Claude smarter in ways that token counting can't capture. There's a third benefit that most users don't think about until they experience it: Claude Code spawns subagents. When you ask Claude to do complex work, it creates child agents — fresh contexts with no memory of your session's history. Without a CLAUDE.md, every subagent starts from zero. With a CLAUDE.md, every subagent starts with your full project context. The instructions you write once are inherited by every agent that touches your project. ### The File Size Problem Every line of CLAUDE.md is injected into the system prompt at the start of each session and each subagent spawn. Longer CLAUDE.md = more tokens consumed before you've typed a word. A 300-line CLAUDE.md costs roughly 3,000 tokens per session. Over 100 sessions, that's 300,000 tokens = about $0.90 on Sonnet. Not much — but that's before you count subagent spawns. If each session spawns three subagents, multiply by four. And a 1,000-line CLAUDE.md is three times as expensive as a 300-line one. The practical guideline: keep your CLAUDE.md under 200 lines. Prefer high-density instructions ("use pytest, not unittest") over verbose explanations ("we use pytest as our test runner because it has better fixture support and more flexible configuration than the built-in unittest module, and most of our team prefers its syntax"). The model understands both — but one is seven times longer. **The 80/20 rule for CLAUDE.md:** 80% of the clarifying questions you'd otherwise answer come from 20% of the possible context. Find those 20% and write them down. Leave everything else out. ### How to Use These Templates Each template below is a starting point. Copy it, delete what doesn't apply, add what's missing. The goal is not a comprehensive document — it's the minimum set of context that saves the maximum number of clarifying questions. **Practical guideline:** Start with the relevant template. Run it through one week of sessions. Note which questions Claude still asks. Add the answers to CLAUDE.md. Delete anything you never reference. Trim to what works. ### The @include Directive Claude Code supports splitting your CLAUDE.md into multiple files using `@include`: ```markdown # Project Instructions @include ./docs/team-standards.md @include ./docs/architecture-decisions.md @include ~/.claude/personal-preferences.md ``` This is useful when: - Team-wide standards should be in a shared file that everyone can update - Personal preferences shouldn't be committed to the project repo - Domain-specific sections get large enough to warrant their own files - Multiple projects share a common base configuration The `@include` directive follows the same priority logic as settings.json: files are loaded in order. Each file is injected at the point of the directive. There's no circular include detection — don't include files that include each other. **Personal style + project context pattern:** ```markdown # CLAUDE.md — Project Name @include ~/.claude/personal-preferences.md @include ./docs/project-context.md ``` Your personal preferences live in `~/.claude/` and follow you across all projects. Project context lives in the repo and is shared with the team. Neither file needs to know about the other. **A note on file size:** The combined token count of all included files counts toward your system prompt. A CLAUDE.md that includes three 200-line files is effectively a 600-line CLAUDE.md. Keep the total lean. --- ## Template 1: Personal Project ### Meet Jordan Jordan is a freelance developer who maintains six side projects simultaneously.
