That is not useful. Line 1, column 47,293 in a bundle could be anything — it is a reference to a location in the optimized, minified output, not the source. The bundle is one enormous line of JavaScript. Position 47,293 in that line could be a database query, a button click handler, or a logging statement. Without the original source code, you cannot tell. Source maps are the solution. A source map is a separate file that maps every position in the compiled bundle back to the corresponding location in the original source. Line 1, column 47,293 in the bundle? The source map says that corresponds to line 892 in `src/QueryEngine.ts`, inside the function `buildRequestContext`, on the call to `normalizeMessages`. With that information, you can look at the original file, understand what is happening on line 892, and debug the problem. Source maps were invented to solve a developer experience problem. When you are debugging your own code in production, you need the source map to trace errors back to readable source. When a customer reports that your application crashed, you get a stack trace with bundle positions. The source map converts those positions to source locations. Without the source map, production debugging is archaeology. Companies publish source maps alongside their compiled code in two different ways, and the difference is critical. The first way: include the source map file in the published package without the source. The source map contains only mappings — position in bundle to position in source file, by name and line number. A developer debugging an integration with your library can use the source map to see that an error occurred in `src/QueryEngine.ts:892`, which tells them the relevant file and line. They cannot see what is on line 892 without the source. The name and location are present. The content is not. The second way: include the actual source files in the source map, embedded directly in the mapping file. The source map becomes self-contained: it has both the mappings and the content they point to. A developer with this source map can see not just that the error was on line 892, but exactly what the code on line 892 says. The second way is often used for development builds, where visibility matters more than size. It is also occasionally used for published public packages when a company wants to give external developers maximum visibility. It is not typically used for closed-source commercial software because it makes the source code available to everyone who installs the package. Most source maps for commercial software use a third variant: they reference external URLs instead of embedding content. The source map points to a location where the source lives. Without access to that location, the source map tells you the name and position, but not the content. This variant has a specific risk: if the URL points to a publicly accessible location, anyone can follow the URL and get the source. Engineers can debug production issues with full source visibility. This is often explicitly desired behavior for internal builds.
